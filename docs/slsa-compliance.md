# SLSA Compliance Status

This document tracks the current compliance status of `@axnic/pi-extension-settings-sdk`
against the [SLSA v1.0](https://slsa.dev/spec/v1.0/) supply-chain security framework,
and lists every gap that must be addressed to reach the next level.

---

## Current Level: **Build L2** (target)

The publish pipeline (`.github/workflows/publish.yml`) uses `npm publish --provenance`
on GitHub-hosted infrastructure, which satisfies SLSA Build L2:

| Requirement | Status | How it is met |
|---|---|---|
| **L1 – Build as code** | ✅ | Build steps are defined in `sdk/package.json` scripts and the GitHub Actions workflow file (`publish.yml`) — no manual build steps. |
| **L1 – Provenance available** | ✅ | `npm publish --provenance` generates a signed SLSA provenance attestation stored on the npm registry and verifiable with `npm audit signatures`. |
| **L2 – Hosted build service** | ✅ | Builds run on GitHub-hosted runners (`ubuntu-latest`), a platform the builder does not control. |
| **L2 – Signed provenance** | ✅ | GitHub Actions mints an OIDC token; npm uses it to countersign the provenance attestation, binding the package to this specific build run, workflow, repo, and commit SHA. |

### Verifying provenance (consumers)

```sh
npm install @axnic/pi-extension-settings-sdk
npm audit signatures
```

## Gaps to Build L3

SLSA Build L3 requires additional hardening of the build platform and the provenance
generation process.  The items below must be addressed — roughly in order of effort —
before L3 can be claimed.

### 1. Pin all GitHub Actions to commit SHAs (high priority)

**Requirement:** Third-party actions used in the build must be pinned to immutable
commit SHAs, not mutable tags (e.g. `actions/checkout@v4` → `actions/checkout@<sha>`).
Mutable tags allow a compromised action to silently change the build environment.

**Gaps in `publish.yml`:**

```yaml
# Current (mutable — not L3 compliant)
uses: actions/checkout@v4
uses: jdx/mise-action@v2
uses: actions/cache@v4

# Required (immutable SHA pins)
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
uses: jdx/mise-action@d5fd3a9d8827df4b0b76d2a4d80d35e9fb8c9c64  # v2.1.9
uses: actions/cache@d4323d4df104b026a6aa633fdb11d772146be0bf   # v4.2.2
```

Apply the same fix to `ci.yml`, `release.yml`, and `security.yml`.

**Tool:** Use [pin-github-action](https://github.com/mheap/pin-github-action) or
Renovate's `github-actions` preset to automate this.

---

### 2. Use `slsa-framework/slsa-github-generator` for provenance (high priority)

**Requirement:** L3 provenance must be generated by a separate, trusted build
service rather than the same workflow that runs the build.  The
[slsa-github-generator](https://github.com/slsa-framework/slsa-github-generator)
reusable workflow separates provenance generation into an isolated, GitHub-controlled
job whose OIDC claims cannot be forged by the builder's workflow.

**What needs to change in `publish.yml`:**

1. Replace the inline `npm publish` step with the
   `slsa-framework/slsa-github-generator/.github/workflows/builder_nodejs_slsa3.yml`
   reusable workflow (currently experimental for Node.js).
2. Pass the built artefact as an input to the generator; let the generator
   publish and attach provenance.

**Blocker:** The Node.js SLSA3 builder from `slsa-framework` is currently
experimental and does not yet support `npm publish --provenance` natively.
Track progress at:
<https://github.com/slsa-framework/slsa-github-generator/issues/409>

---

### 3. Restrict network access during the build (medium priority)

**Requirement:** L3 build environments should not have unrestricted outbound
network access — a compromised dependency could exfiltrate secrets or alter the
output.

**What needs to change:**

- Use a network-policy-aware runner (e.g. GitHub's larger runners with
  network egress controls, or a self-hosted runner with an egress firewall).
- Alternatively, pre-fetch all dependencies in a separate, audited step and
  run the compilation step with network disabled.

**Current state:** Standard `ubuntu-latest` runners have unrestricted outbound
network access during `npm ci` and `npm run build`.

---

### 4. Ensure reproducible builds (medium priority)

**Requirement:** Given identical inputs (source commit + dependency lock file),
the build must produce a byte-for-byte identical artefact.

**What needs to change:**

- Audit the SDK build for any non-deterministic outputs (timestamps embedded in
  generated files, non-stable sort orders in `tsc` output, etc.).
- Consider using `SOURCE_DATE_EPOCH` to normalise file timestamps.
- Add a CI step that builds twice and diffs the two artefacts.

**Current state:** Not verified.  `tsc` output is generally deterministic but
has not been tested.

---

### 5. Protect the default branch with required PR reviews (low priority)

**Requirement:** Code that reaches the build must have been reviewed by a party
other than the author, to prevent a single compromised identity from introducing
malicious changes.

**What needs to change:**

- Enable "Require a pull request before merging" on the default branch
  (Settings → Branches → Branch protection rules).
- Set minimum 1 required approving review.
- Enable "Dismiss stale pull request approvals when new commits are pushed".

**Note:** This also blocks the `release.yml` workflow from pushing the version
bump commit directly.  A PAT or the
[`RELEASE_TOKEN` workaround](./workflows/release.yml) is needed once branch
protection is active.

---

### 6. Enable Dependabot for automated dependency updates (low priority)

Keeping dependencies current reduces the attack surface audited by `npm audit`
(see `security.yml`).

Add `.github/dependabot.yml`:

```yaml
version: 2
updates:
  - package-ecosystem: npm
    directory: "/"
    schedule:
      interval: weekly
  - package-ecosystem: npm
    directory: "/sdk"
    schedule:
      interval: weekly
  - package-ecosystem: github-actions
    directory: "/"
    schedule:
      interval: weekly
```
